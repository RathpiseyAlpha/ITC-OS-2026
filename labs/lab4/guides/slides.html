I/O Redirection

Control where command output goes — stdout, stderr, and stdin

04

The Three Standard Streams

stdin (0)  →  [Command]  →  stdout (1)
                                             ↘
                                         stderr (2)
  • stdin (fd 0) — input stream (keyboard by default)
  • stdout (fd 1) — normal output (terminal by default)
  • stderr (fd 2) — error output (terminal by default)
Redirection operators let you send each stream to a file, discard it, or merge them together.

I/O REDIRECTION 6 / 30

Redirection in Action

$ ls /etc | head -n 10 > etc_listing.txt # stdout to file $ ls /nonexistent 2> errors.txt # stderr to file $ ls /etc /nonexistent > out.txt 2> err.txt # separate streams $ ls /etc /nonexistent &> combined.txt # both to one file $ sort < fruits.txt # stdin from file $ ls /etc > /dev/null # discard stdout
Caution: > overwrites the target file. Use >> to append. Always check before overwriting important files!

Pipelines & Filters

Chain commands together — one command's output flows into the next

08

How Pipes Work

[cmd1]  →  stdout  |  stdin  →  [cmd2]  →  stdout  |  stdin  →  [cmd3]
  • The pipe | connects stdout of one command to stdin of the next
  • Commands run concurrently — data flows through as it’s produced
  • Only stdout passes through pipes; stderr still goes to the terminal
  • Use |& to pipe both stdout and stderr (Bash 4+)

Data Analysis

grep, cut & awk — the Swiss Army knives of text processing

12

grep — Pattern Searching

  • grep "pattern" file — show lines matching a pattern
  • grep -v "pattern" — invert match (lines NOT matching)
  • grep -c "pattern" — count matching lines
  • grep -i "pattern" — case-insensitive search
  • grep -E "regex" — extended regex (egrep)
  • grep -r "pattern" dir/ — recursive search
$ grep "GET" access.log # all GET requests $ grep -v '" 200 ' access.log # non-200 status codes $ grep -c "POST" access.log # count POST requests $ grep -E '" (40[0-9]) ' access.log # all 4xx errors
DATA ANALYSIS 13 / 30

cut & awk — Field Extraction

cut

$ cut -d: -f1 /etc/passwd $ cut -d' ' -f1 access.log $ cut -d: -f1,7 /etc/passwd

Splits by delimiter, extracts fields by number.

awk

$ awk '{print $1, $9}' access.log $ awk '{sum += $10} END {print sum}' $ awk -F: '$3 >= 1000' /etc/passwd

More powerful: conditional logic, math, and formatting.

When to use which: cut for simple delimited fields. awk when you need logic, math, or complex field extraction.

Combined Analysis Pipelines

$ awk '{print $1}' access.log | sort | uniq -c | sort -rn # Requests per IP address, sorted by frequency $ awk '{print $7}' access.log | sort | uniq -c | sort -rn # Top requested URLs $ grep "/bin/bash" /etc/passwd | cut -d: -f1 # Users with bash shell $ awk -F: '$3 < 1000 {count++} END {print "System users:", count}' /etc/passwd # Count system vs regular users
Power pattern: extract | sort | uniq -c | sort -rn — the universal frequency analysis pipeline.

Job Control

[Foreground] —Ctrl+Z—→ [Stopped] —bg—→ [Background]
                                               ↑
[Background] —fg—→ [Foreground]
  • command & — start in background
  • Ctrl+Z — suspend foreground process
  • jobs -l — list jobs with PIDs
  • fg %1 — bring job 1 to foreground
  • bg %1 — resume job 1 in background
  • nohup cmd & — run immune to terminal close

Orphan & Zombie Processes

What happens when parent processes die or children go unreaped?

20

Orphan Processes

Parent (exits)  →  Child still running  →  Adopted by init (PID 1)
  • An orphan is a child process whose parent has terminated
  • The init / systemd process (PID 1) adopts the orphan
  • The child’s PPID changes to 1 — verify with ps -o pid,ppid,cmd
  • Orphans continue running normally and are cleaned up when they exit
Key observation: Run the orphan program, then quickly check with ps — the child’s PPID should be 1.
ORPHAN & ZOMBIE 21 / 30

Zombie Processes

Child exits  →  Parent doesn't call wait()  →  Zombie (Z / <defunct>)
  • A zombie is a child that has exited but whose parent hasn’t read its exit status
  • The process entry remains in the process table (state Z)
  • Shows as <defunct> in ps output
  • Cannot be killed with kill -9 — it’s already dead!
  • Cleaned up when the parent calls wait() or when the parent exits
Danger: Too many zombies can exhaust the process table. The fix is to fix the parent (add wait()), not to try killing the zombie.
