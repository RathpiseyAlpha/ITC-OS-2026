Wildcards

Pattern matching for efficient file management

Wildcard Patterns

PatternSymbolMeaningExample
Match any*Zero or more characters*.txt → all .txt files
Single char?Exactly one characterfile?.txt → file1.txt
Char set[abc]One from setfile[123].txt
Char range[a-z]One in range[A-Z]*.txt
Negate[!abc]NOT in setfile[!0-9].txt
Braces{a,b}Generate multiplefile.{txt,log}
Key distinction: [...] matches one character from a set. {...} generates multiple separate strings (brace expansion happens before globbing).
WILDCARDS 5 / 28

Wildcards in Action

  • List all text files: ls *.txt
  • List reports 01–03 only: ls report0[1-3].txt
  • Skip January logs: ls log_[!j]*.log
  • Create 5 files at once: touch memo_{mon,tue,wed,thu,fri}.txt
  • Copy all CSVs: cp *.csv archive/
  • Remove all temp files: rm *.tmp
Caution: Always double-check rm with wildcards! Use ls *.tmp first to preview what will be deleted.

Links

Hard links & symbolic links — the difference matters

07

Understanding Inodes

  • Every file is identified by an inode — a unique number storing metadata (permissions, size, disk location)
  • The filename is just a directory entry pointing to an inode
  • Multiple names can point to the same inode → hard links
  • A file’s data is deleted only when the last hard link is removed
filename_A  →  inode 12345  →  [disk data]
filename_B  →  inode 12345  →  [same data!]
symlink_C  →  inode 99999  →  "path: filename_A"

GRUB & Boot Process

From power-on to login prompt

11

Linux Boot Sequence

BIOS/UEFI → GRUB → Kernel → init/systemd → Login
  • BIOS/UEFI: Power-On Self Test (POST), finds boot device
  • GRUB: Loads from boot device, presents menu, loads kernel
  • Kernel: Initializes hardware, mounts root FS, starts first process (PID 1)
  • init/systemd: Starts system services in correct order
  • Login: User space ready

SHARED LIBS 17 / 28

How the Dynamic Linker Works

Program starts → Kernel loads ELF → ld-linux.so (dynamic linker)
↓
Reads program headers → Finds required .so files
↓
Search order: LD_LIBRARY_PATH → /etc/ld.so.cache → /lib, /usr/lib
↓
Maps libraries into process address space → Program runs!
SHARED LIBS 18 / 28

Shared Library Commands

  • ldd /bin/ls — list all shared library dependencies
  • ldconfig -p | grep libc — search the library cache
  • sudo ldconfig — refresh the cache after installing a new .so
  • file /lib/x86_64-linux-gnu/libc.so.6 — identify file type (ELF)
  • readelf -h /bin/ls — examine ELF headers
  • cat /proc/$$/maps | grep '.so' — show loaded libraries in current shell
SHARED LIBS 19 / 28

Building a Custom Shared Library

# Step 1: Compile to position-independent object code gcc -fPIC -c mylib.c -o mylib.o # Step 2: Link into a shared library gcc -shared -o libmylib.so mylib.o # Step 3: Install system-wide sudo cp libmylib.so /usr/local/lib/ sudo ldconfig # Step 4: Compile a program using the library gcc main.c -L/usr/local/lib -lmylib -o myprogram
-fPIC = Position-Independent Code (required for .so files so they can load at any address)
